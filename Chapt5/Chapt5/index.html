    几个原则

    @startmindmap
+ **面向对象设计原则**
++ 单一职责原则
+++_ 就一个类别而言，应该仅有一个引起他变化的原因
++ 开闭原则
+++_ 对拓展开放，对修改关闭
++++_ 类特征的抽象层相对稳定，不需要更改
++++_ 从抽象类导出的具体类可以改变系统行为
++ 里氏代换原则
+++_ 一个软件实体如果使用的是一个基类的话，那么一的也适用于其子类
++ 依赖倒转原则
+++_ 高层模块不应该依赖低层模块，他们都应该依赖抽象，要针对接口编程

++ 接口隔离原则
+++_ 客户端不应该依赖他不需要的接口“一个类对另一个类的依赖应该建立在最小的接口上
++ 合成复用原则
+++_ 尽量使用对象组合，而不是继承达到复用的目的，即**首先采用合成/聚合，其次才考虑继承**
++ 迪米特法则
+++_ 一个软件实体应当尽可能减少与其他实体发生相互作用
@endmindmap

    单例模式

    保证一个类仅仅有一个实例，并提供一个访问他的方法

    几种方法
    饿汉式

    在类加载的时候会立刻实例化

    public class Single{
    private static Single ele = new SIngle();
    private Sinngle(){
    }
    public static Single getInstance(){
        return eke;
    }
}
    懒汉式

    在类加载的时候没有直接初始化，而是在调用指定实例化方法的时候在进行实例。这样能保证不想使用的时候也不会实例化，一般比饿汉式效率高。

    public class Single{
    private static Single ele;
    private Sinngle(){
    }
    public static Single getInstance(){
        if(ele == null){
            ele = new Single();
        }
        return ele;
    }
}
    同步锁

    添加synchronized关键字到getInstance()方法中，迫使每个线程在进入该方法前，要先等候别的线程离开方法，也就是说，不会有两个线程同时进入这个方法

    public class Single{
    private static Single ele;
    private Sinngle(){
    }
    public static synchronized Single getInstance(){
        if(ele == null){
            ele = new Single();
        }
        return ele;
    }
}
    总结

    工厂模式

    简单工厂模式：并不是一个真正意义上的设计模式

    不符合开闭原则:每次添加新产品就需要修改工厂类 整个工厂在一个class类里

    工厂方法模式：分为产品线和工厂线

    每增加新的产品，只需要增加该产品以及对应的具体实现工厂类$\rightarrow$符合开闭原则：拓展时不需修改原来的代码

    抽象工厂模式

    便于拓展一个产品族（品牌），但是拓展一个产品类型的时候对顶层工厂类的接口进行修改
